M?dulo 01 ? 10 min

Introdução ao Java Moderno 8→21

Java não ficou só mais completo: ficou estrategicamente mais poderoso. Este arranque organiza o teu mapa mental para avançares com confiança técnica desde o primeiro módulo.

Objetivos de Aprendizagem

  • Perceber a evolução do Java do paradigma clássico para um estilo híbrido OO+funcional.
  • Reconhecer os blocos de funcionalidades que mais impactam produtividade e legibilidade.
  • Definir um critério prático para decidir quando adotar sintaxe moderna.

Explica??o por Blocos

Do Java verboso ao Java intencional

Até Java 7, muita lógica útil ficava escondida em ruído sintático. Desde o Java 8, a linguagem reduz cerimónia sem abdicar de tipagem forte.

A mudança não é cosmética: altera a forma de modelar pipelines de dados, APIs e domínios.

Três ondas de modernização

Java 8 trouxe Lambdas, Streams, Optional e melhorias em Collections.

Java 9–16 consolidou módulos, API HTTP, var, text blocks e switch expressions.

Java 17–21 reforçou modelação de domínio e concorrência com records, sealed classes, pattern matching e virtual threads.

Como usar este percurso

Segue a ordem recomendada. Cada módulo depende parcialmente do anterior.

Foca em três perguntas por tema: quando usar, quando evitar e como validar impacto no código real.

Exemplos Comentados (Antes vs Moderno)

Mudança de estilo: de loops a pipelines

Antes

List<String> resultado = new ArrayList<>();
for (String nome : nomes) {
  if (nome.startsWith("A")) {
    resultado.add(nome.toUpperCase());
  }
}

Moderno

List<String> resultado = nomes.stream()
  .filter(nome -> nome.startsWith("A"))
  .map(String::toUpperCase)
  .toList();

A intenção fica explícita: filtrar, transformar, materializar. Menos controlo manual, mais foco na regra.

Erros Comuns e Anti-padr?es

  • Adotar features novas sem um critério claro de legibilidade e manutenção.
  • Misturar estilos antigo e moderno no mesmo bloco, dificultando leitura.
  • Ignorar medição de impacto em performance e não considerar contexto do domínio.

Checkpoint (Mini Quiz)

1. Qual é o principal ganho ao migrar para idioms modernos do Java?

  1. Reduzir memória usada em qualquer cenário.
  2. Expressar intenção com menos ruído sintático e melhor legibilidade. (correta)
  3. Eliminar totalmente a necessidade de orientação a objetos.
  4. Substituir todas as APIs antigas automaticamente.

Explica??o: O ganho central é clareza e expressividade. Performance depende do contexto e do desenho.

2. A sequência mais consistente para estudar no manual é:

  1. Começar por virtual threads e voltar para Lambdas no fim.
  2. Começar por módulos Java 9 e ignorar Java 8.
  3. Começar por Java 8 (Lambdas/Streams) e evoluir para versões posteriores. (correta)
  4. Estudar apenas var e records porque cobrem todas as mudanças.

Explica??o: Os conceitos de Java 8 são a base de boa parte das melhorias seguintes.

Recap em 5 Pontos

  1. Java moderno é evolução de estilo, não ruptura total.
  2. Adoção gradual funciona melhor do que refatoração cega.
  3. Java 8 é base; Java 9–21 consolida capacidade e ergonomia.
  4. Legibilidade e intenção devem guiar decisões.
  5. Cada módulo seguinte traduz este princípio em prática.

M?dulo 02 ? 14 min

Expressões Lambda

Aqui começa a viragem real: menos ruído, mais intenção. Lambdas dão-te velocidade sem abdicar de tipagem forte.

Objetivos de Aprendizagem

  • Identificar interfaces funcionais e mapear comportamento para lambdas.
  • Converter classes anónimas em sintaxe lambda de forma segura.
  • Evitar lambdas excessivas que reduzem legibilidade.

Explica??o por Blocos

Sintaxe mínima com semântica clara

Uma lambda representa implementação inline de um único método abstrato.

Padrão: (parâmetros) -> expressão ou bloco.

Onde aplicar com maior retorno

Callbacks, filtros, ordenação e transformação de dados.

Integração forte com Streams e method references.

Quando recuar

Se a lambda cresce demasiado, extrai para método nomeado.

Nomear intenção continua a ser obrigatório quando o comportamento é complexo.

Exemplos Comentados (Antes vs Moderno)

Thread: classe anónima vs lambda

Antes

Runnable tarefa = new Runnable() {
  @Override
  public void run() {
    System.out.println("Processar lote...");
  }
};
new Thread(tarefa).start();

Moderno

Runnable tarefa = () -> System.out.println("Processar lote...");
new Thread(tarefa).start();

A implementação fica focada no comportamento essencial do callback.

Comparator com menos ruído

Antes

Collections.sort(produtos, new Comparator<Produto>() {
  @Override
  public int compare(Produto a, Produto b) {
    return Double.compare(a.getPreco(), b.getPreco());
  }
});

Moderno

produtos.sort((a, b) -> Double.compare(a.getPreco(), b.getPreco()));

A operação de ordenação fica num único ponto sem boilerplate de classe anónima.

Erros Comuns e Anti-padr?es

  • Usar lambdas longas com múltiplas responsabilidades.
  • Forçar efeitos colaterais dentro de pipelines funcionais.
  • Esquecer que a interface alvo tem de ser funcional.

Checkpoint (Mini Quiz)

1. Uma lambda em Java exige que o tipo alvo seja:

  1. Uma classe abstrata com dois métodos.
  2. Uma interface funcional com um método abstrato. (correta)
  3. Qualquer enum com método estático.
  4. Uma classe concreta com construtor vazio.

Explica??o: O compilador precisa de um contrato funcional único para inferir o tipo.

2. Qual prática melhora legibilidade com lambdas?

  1. Colocar toda a lógica de negócio dentro da lambda.
  2. Usar lambdas mesmo quando o método nomeado seria mais claro.
  3. Extrair para método quando o bloco deixa de ser curto e óbvio. (correta)
  4. Evitar interfaces funcionais e usar Object.

Explica??o: Lambdas brilham em comportamento curto. Fluxos complexos pedem nomeação explícita.

Recap em 5 Pontos

  1. Lambdas reduzem boilerplate de classes anónimas.
  2. Interface funcional é pré-requisito técnico.
  3. Curto e expressivo ganha de longo e opaco.
  4. Method references são evolução natural quando a lambda só delega.
  5. Streams tornam-se muito mais poderosos com lambdas bem desenhadas.

M?dulo 03 ? 13 min

Interfaces no Java Moderno (8→17)

Interfaces deixaram de ser só contrato. Neste módulo, vais dominar como elas passaram a carregar comportamento útil e modelação segura.

Objetivos de Aprendizagem

  • Aplicar métodos default e static para evoluir APIs sem quebrar implementações antigas.
  • Resolver conflitos de métodos default com override explícito e Interface.super.metodo().
  • Usar private methods (Java 9) e sealed interfaces (Java 17) com critério de design.

Explica??o por Blocos

Java 8: default + static em interfaces

default methods permitem adicionar comportamento sem forçar implementação imediata em classes existentes.

static methods na interface concentram utilitários do próprio domínio, evitando classes auxiliares genéricas.

Conflitos de default methods

Quando duas interfaces oferecem o mesmo método default, a classe concreta precisa resolver explicitamente o conflito.

A resolução pode usar implementação própria ou delegar para uma interface específica com Interface.super.metodo().

Java 9 e Java 17

Java 9 introduz private methods para reutilização interna entre default/static, reduzindo duplicação no contrato.

Java 17 introduz sealed interfaces para limitar implementações permitidas com permits, sealed, final e non-sealed.

Interfaces e pattern matching

Hierarquias seladas tornam o tratamento por tipo mais explícito e previsível.

Em versões recentes, a combinação com switch por padrões melhora clareza em regras de domínio.

Exemplos Comentados (Antes vs Moderno)

Interface evolutiva com default, static e private

Antes

interface Notificador {
  void enviar(String mensagem);
}

Moderno

interface Notificador {
  default String prefixo() {
    return "[APP]";
  }

  static boolean emailValido(String email) {
    return email != null && email.contains("@");
  }

  default void info(String mensagem) {
    log("INFO", mensagem);
  }

  private void log(String nivel, String mensagem) {
    System.out.println(prefixo() + " " + nivel + ": " + mensagem);
  }
}

A interface mantém contrato e passa a incluir comportamento reutilizável sem quebrar versões anteriores.

Sealed interface com implementações controladas

Antes

interface Resultado {}
class Sucesso implements Resultado {}
class Falha implements Resultado {}

Moderno

sealed interface Resultado permits Sucesso, Falha {}

final class Sucesso implements Resultado {}
non-sealed class Falha implements Resultado {}
final class FalhaValidacao extends Falha {}

static String resumo(Resultado r) {
  return switch (r) {
    case Sucesso s -> "Operação concluída";
    case FalhaValidacao fv -> "Falha de validação";
    case Falha f -> "Falha genérica";
  };
}

Com sealed interfaces, o conjunto de implementações passa a ser explícito e mais seguro para manutenção.

Erros Comuns e Anti-padr?es

  • Usar default methods para lógica pesada que deveria viver em serviço de domínio.
  • Ignorar conflitos entre interfaces e deixar resolução implícita.
  • Declarar sealed interface sem desenhar claramente as implementações permitidas.

Checkpoint (Mini Quiz)

1. Qual vantagem principal dos métodos default em interfaces?

  1. Substituir classes concretas em todos os cenários.
  2. Evoluir interfaces mantendo compatibilidade com implementações existentes. (correta)
  3. Eliminar necessidade de testes unitários.
  4. Forçar herança múltipla de classes.

Explica??o: default methods permitem adicionar comportamento sem quebrar classes já implementadas.

2. Como resolver conflito de default methods de duas interfaces?

  1. Não é possível, o compilador escolhe automaticamente.
  2. Remover uma das interfaces do código.
  3. Sobrescrever o método e, se necessário, usar Interface.super.metodo(). (correta)
  4. Converter as interfaces em classes abstratas.

Explica??o: A classe concreta deve resolver explicitamente o conflito.

3. Qual novidade do Java 17 em interfaces é correta?

  1. Interfaces passaram a permitir construtores públicos.
  2. Sealed interfaces permitem restringir implementações com permits. (correta)
  3. Interfaces deixaram de aceitar métodos static.
  4. Métodos private foram removidos.

Explica??o: sealed interfaces tornam a hierarquia explícita e controlada.

Recap em 5 Pontos

  1. default methods ajudam na evolução de API sem ruptura.
  2. static methods mantêm utilitários coesos na própria interface.
  3. Conflitos de default exigem resolução explícita.
  4. private methods reduzem duplicação interna da interface.
  5. sealed interfaces melhoram segurança e clareza de domínio.

M?dulo 04 ? 18 min

API de Streams e combinação map/reduce

Se queres escrever código que parece engenharia e não burocracia, Streams é o salto. Vais pensar em fluxo de dados, não em controlo manual.

Objetivos de Aprendizagem

  • Separar operações intermédias e terminais com precisão.
  • Construir pipelines com filter/map/reduce sem efeitos colaterais.
  • Avaliar quando usar stream sequencial vs paralelo.

Explica??o por Blocos

Modelo mental de pipeline

Origem de dados -> operações intermédias -> operação terminal.

Intermédias são lazy; execução real só acontece na terminal.

map/reduce em problemas reais

map transforma cada elemento para uma nova representação.

reduce agrega toda a sequência num único resultado (soma, máximo, composição).

Paralelismo com cautela

parallelStream pode escalar em cargas CPU-bound, mas exige operações associativas e sem estado partilhado.

Sem esses cuidados, o ganho de throughput pode virar regressão ou resultado incorreto.

Exemplos Comentados (Antes vs Moderno)

Pipeline de limpeza e agregação

Antes

int total = 0;
for (Pedido p : pedidos) {
  if (p.estaPago()) {
    total += p.getItens().size();
  }
}

Moderno

int total = pedidos.stream()
  .filter(Pedido::estaPago)
  .mapToInt(p -> p.getItens().size())
  .sum();

O pipeline descreve o processo em etapas sem controlar manualmente índice e acumulador.

Map + Reduce

Antes

double soma = 0.0;
for (Produto produto : produtos) {
  soma += produto.getPreco() * produto.getQuantidade();
}

Moderno

double soma = produtos.stream()
  .map(produto -> produto.getPreco() * produto.getQuantidade())
  .reduce(0.0, Double::sum);

Primeiro transformas cada item em valor monetário. Depois reduzes para total.

Erros Comuns e Anti-padr?es

  • Usar forEach para mutar estado externo em vez de coletores apropriados.
  • Executar operações pesadas em parallelStream sem benchmark.
  • Confundir map (transformação) com peek (observação/debug).

Checkpoint (Mini Quiz)

1. Qual afirmação está correta sobre streams?

  1. Operações intermédias executam sempre de imediato.
  2. A execução ocorre quando existe operação terminal. (correta)
  3. Streams alteram obrigatoriamente a coleção original.
  4. parallelStream garante ordem de processamento.

Explica??o: A execução é lazy até à operação terminal.

2. Em map + reduce, o map deve:

  1. Agregar resultados num único valor.
  2. Parar a execução do stream.
  3. Transformar cada elemento para novo valor. (correta)
  4. Ordenar elementos antes de reduzir.

Explica??o: A agregação é papel do reduce; map apenas converte elemento a elemento.

3. Quando parallelStream tende a ser opção razoável?

  1. Em listas pequenas com lógica leve.
  2. Quando há estado mutável partilhado.
  3. Em cargas pesadas e operações independentes após benchmark. (correta)
  4. Quando é obrigatório manter ordem exata com forEach.

Explica??o: Paralelismo só compensa quando custo de trabalho justifica overhead e não há dependências perigosas.

Recap em 5 Pontos

  1. Streams introduzem estilo declarativo para coleções.
  2. Intermédias são lazy; terminal dispara execução.
  3. map transforma; reduce agrega.
  4. Evita mutação externa dentro do pipeline.
  5. Paralelismo exige medição e desenho seguro.

M?dulo 05 ? 12 min

Referências a Métodos e Construtores

Este módulo é sobre elegância com critério: quando o método já existe, referências limpam o código e aumentam leitura imediata.

Objetivos de Aprendizagem

  • Distinguir os quatro tipos principais de referências.
  • Converter lambdas delegadoras para referências equivalentes.
  • Escolher entre lambda e referência com critério de clareza.

Explica??o por Blocos

Os quatro formatos

Classe::metodoEstatico, objeto::metodoInstancia, Classe::metodoInstancia e Classe::new.

Todos preservam o mesmo contrato funcional da lambda equivalente.

Vantagem real

Melhor leitura quando o nome do método já comunica intenção.

Ótimo para pipelines e operações em coleções.

Limite prático

Se precisas de adaptar argumentos, lógica condicional ou múltiplas chamadas, lambda explícita costuma ser melhor.

Exemplos Comentados (Antes vs Moderno)

Impressão direta

Antes

nomes.forEach(nome -> System.out.println(nome));

Moderno

nomes.forEach(System.out::println);

A referência reduz ruído sem alterar comportamento.

Construção de objetos em stream

Antes

List<Aluno> alunos = nomes.stream()
  .map(nome -> new Aluno(nome))
  .toList();

Moderno

List<Aluno> alunos = nomes.stream()
  .map(Aluno::new)
  .toList();

Construtores também podem ser referenciados quando a assinatura coincide.

Erros Comuns e Anti-padr?es

  • Usar referência só para parecer moderno, mesmo quando piora compreensão.
  • Confundir Classe::metodoInstancia com método estático.
  • Perder contexto do argumento implícito (receiver) em streams.

Checkpoint (Mini Quiz)

1. Qual opção representa referência a construtor?

  1. Produto::criar
  2. new Produto::nome
  3. Produto::new (correta)
  4. Produto.new()

Explica??o: A sintaxe correta para construtor é Classe::new.

2. Quando preferir lambda em vez de method reference?

  1. Quando só encaminha para um método já pronto.
  2. Quando requer lógica adicional para adaptar dados. (correta)
  3. Quando vais usar forEach simples.
  4. Quando queres reduzir verbosidade ao máximo sempre.

Explica??o: Se há transformação ou lógica extra, a lambda explicita melhor o comportamento.

Recap em 5 Pontos

  1. Method references são forma concisa de lambdas delegadoras.
  2. Há quatro padrões de referência principais.
  3. Clareza manda na escolha entre lambda e referência.
  4. Construtores integram bem com map.
  5. Evita dogma: usa o que comunica melhor.

M?dulo 06 ? 14 min

Classe Optional

Optional não é moda: é clareza de contrato. Vais reduzir ambiguidade e tornar o teu código muito mais previsível.

Objetivos de Aprendizagem

  • Modelar ausência de valor explicitamente com Optional.
  • Usar map/flatMap/orElse/orElseGet/orElseThrow em cadeia segura.
  • Evitar abuso de Optional em campos e parâmetros inadequados.

Explica??o por Blocos

Contrato explícito de ausência

Optional comunica ao consumidor que o valor pode não existir.

Substitui verificações manuais repetitivas por um fluxo explícito.

Operadores centrais

map transforma o valor presente; flatMap evita Optional aninhado.

orElseGet é preferível a orElse quando o fallback é caro.

Boas fronteiras

Optional funciona bem em retornos de método.

Evita usar em atributos de entidades persistentes ou em argumentos obrigatórios.

Exemplos Comentados (Antes vs Moderno)

Acesso seguro a valor opcional

Antes

String nome = null;
if (utilizador != null && utilizador.getPerfil() != null) {
  nome = utilizador.getPerfil().getNome();
}
if (nome == null) {
  nome = "Visitante";
}

Moderno

String nome = Optional.ofNullable(utilizador)
  .map(Utilizador::getPerfil)
  .map(Perfil::getNome)
  .orElse("Visitante");

A cadeia expressa claramente o percurso até ao valor e o fallback.

Fallback eficiente

Antes

String token = optionalToken.isPresent()
  ? optionalToken.get()
  : gerarTokenPesado();

Moderno

String token = optionalToken.orElseGet(this::gerarTokenPesado);

orElseGet evita calcular fallback quando o valor já existe.

Erros Comuns e Anti-padr?es

  • Usar Optional#get sem validar presença.
  • Devolver null dentro de Optional, quebrando o contrato.
  • Aplicar Optional em campos de DTO/entidade sem necessidade.

Checkpoint (Mini Quiz)

1. Qual método evita executar fallback pesado desnecessariamente?

  1. orElse
  2. get
  3. orElseGet (correta)
  4. ifPresent

Explica??o: orElseGet recebe Supplier e executa apenas se Optional estiver vazio.

2. Uso mais recomendado de Optional é em:

  1. Parâmetros obrigatórios de método.
  2. Retornos onde ausência é válida. (correta)
  3. Todos os atributos de entidades JPA.
  4. Constantes estáticas.

Explica??o: Retorno opcional deixa explícito o contrato de ausência sem poluir modelos.

Recap em 5 Pontos

  1. Optional torna ausência de valor explícita.
  2. map/flatMap evitam ifs aninhados.
  3. orElseGet é melhor para fallback caro.
  4. Optional#get direto é fonte de erro.
  5. Usa Optional com critério de design de API.

M?dulo 07 ? 13 min

Melhorias na API de Coleções (Java 8)

Aqui transformas operações comuns em código direto e confiante. Coleções modernas dão-te produtividade sem sacrificar controlo.

Objetivos de Aprendizagem

  • Aplicar forEach, removeIf, replaceAll e computeIfAbsent corretamente.
  • Combinar Collections com Streams sem mutações perigosas.
  • Distinguir coleções mutáveis de imutáveis em APIs.

Explica??o por Blocos

Operações in-place mais legíveis

Métodos novos evitam loops mecânicos para tarefas comuns.

A semântica fica no nome da operação, não no controlo manual.

Mapas mais inteligentes

computeIfAbsent simplifica inicialização condicional de listas e caches.

merge e compute ajudam em agregações incrementais.

Imutabilidade como padrão de saída

Quando devolves dados para consumo externo, coleções imutáveis reduzem bugs e efeitos colaterais.

Exemplos Comentados (Antes vs Moderno)

removeIf em vez de loop manual

Antes

Iterator<String> it = tags.iterator();
while (it.hasNext()) {
  if (it.next().isBlank()) {
    it.remove();
  }
}

Moderno

tags.removeIf(String::isBlank);

Código curto e intenção imediata.

computeIfAbsent para indexar por chave

Antes

if (!indice.containsKey(categoria)) {
  indice.put(categoria, new ArrayList<>());
}
indice.get(categoria).add(produto);

Moderno

indice.computeIfAbsent(categoria, c -> new ArrayList<>())
  .add(produto);

A inicialização preguiçosa da coleção fica encapsulada na API do Map.

Erros Comuns e Anti-padr?es

  • Misturar mutação direta e streams no mesmo fluxo sem necessidade.
  • Assumir que List.of (Java 9+) é mutável.
  • Usar computeIfAbsent com funções que têm efeitos colaterais inesperados.

Checkpoint (Mini Quiz)

1. Qual método do Map evita o padrão containsKey + put?

  1. replaceAll
  2. computeIfAbsent (correta)
  3. forEach
  4. sorted

Explica??o: computeIfAbsent cria valor apenas quando a chave não existe.

2. Qual é uma vantagem de devolver coleção imutável em API?

  1. Aumenta automaticamente performance em todos os cenários.
  2. Impede mudanças externas acidentais no estado. (correta)
  3. Permite adicionar elementos sem validação.
  4. Substitui testes unitários.

Explica??o: Imutabilidade limita efeitos colaterais e facilita manutenção.

Recap em 5 Pontos

  1. Java 8 trouxe operações diretas para coleções e mapas.
  2. removeIf/replaceAll reduzem loops técnicos.
  3. computeIfAbsent simplifica inicialização por chave.
  4. Coleções imutáveis ajudam a proteger contratos de API.
  5. Semântica clara vale mais do que micro-otimização prematura.

M?dulo 08 ? 9 min

Scanner, Warnings e try-with-resources

Detalhes de recursos separam código robusto de código frágil. Este módulo afina decisões que evitam bugs silenciosos em produção.

Objetivos de Aprendizagem

  • Distinguir cenários onde Scanner deve ser fechado explicitamente.
  • Aplicar try-with-resources em fontes que exigem ciclo de vida controlado.
  • Evitar fechar System.in sem intenção explícita.

Explica??o por Blocos

Fechar quando há recurso real

Scanner sobre ficheiros, sockets ou streams dedicadas deve ser fechado.

try-with-resources reduz erro humano e deixa ciclo de vida explícito.

System.in é caso especial

Fechar Scanner ligado a System.in fecha também a entrada padrão.

Em aplicações interativas, isso pode quebrar leitura posterior.

Política de projeto

Define convenção: quem cria recurso, fecha recurso.

Documenta exceções no código quando decides não fechar.

Exemplos Comentados (Antes vs Moderno)

Leitura de ficheiro com fechamento seguro

Antes

Scanner sc = new Scanner(new File("dados.txt"));
while (sc.hasNextLine()) {
  processar(sc.nextLine());
}
sc.close();

Moderno

try (Scanner sc = new Scanner(new File("dados.txt"))) {
  while (sc.hasNextLine()) {
    processar(sc.nextLine());
  }
}

try-with-resources garante close mesmo com exceções.

System.in com cuidado

Antes

Scanner sc = new Scanner(System.in);
String nome = sc.nextLine();
sc.close(); // pode encerrar entrada global

Moderno

Scanner sc = new Scanner(System.in);
String nome = sc.nextLine();
// não fechar aqui se a aplicação continuar a ler do teclado

A decisão depende do ciclo de vida global da aplicação.

Erros Comuns e Anti-padr?es

  • Ignorar warnings sem avaliar tipo de recurso.
  • Fechar System.in cedo e quebrar fluxos seguintes.
  • Duplicar responsabilidades de fecho em camadas diferentes.

Checkpoint (Mini Quiz)

1. Quando try-with-resources é especialmente recomendado?

  1. Scanner sobre ficheiro ou stream dedicado. (correta)
  2. Sempre que houver variável local de texto.
  3. Quando queres evitar usar exceções.
  4. Apenas em métodos estáticos.

Explica??o: Recursos reais devem ter ciclo de vida explícito e seguro.

2. Qual risco de fechar Scanner ligado a System.in?

  1. Nenhum, é sempre obrigatório fechar.
  2. Apenas perda de performance.
  3. Encerrar entrada padrão e bloquear futuras leituras. (correta)
  4. Mudar encoding default da JVM.

Explica??o: Fechar System.in afeta toda a aplicação.

Recap em 5 Pontos

  1. Warnings de Scanner exigem análise de contexto.
  2. try-with-resources é padrão seguro para recursos dedicados.
  3. System.in deve ser tratado com cautela.
  4. Responsabilidade de fecho deve ser clara entre camadas.
  5. Boa política evita leaks e bugs de I/O.

M?dulo 09 ? 8 min

Operador Diamond (<>)

Pequena sintaxe, grande efeito cumulativo. Diamond remove fricção diária e deixa a intenção genérica mais limpa.

Objetivos de Aprendizagem

  • Aplicar diamond em construções genéricas seguras.
  • Identificar limites de inferência em contextos complexos.
  • Evitar ambiguidades que degradam legibilidade.

Explica??o por Blocos

Redução de repetição

Antes do Java 7, era comum duplicar o tipo genérico na declaração e no construtor.

Com <>, o compilador infere tipos a partir do lado esquerdo.

Benefícios práticos

Menos ruído visual e menor probabilidade de inconsistência entre tipos.

Melhor leitura em estruturas aninhadas de coleções.

Restrições e clareza

Há casos onde declarar tipo explicitamente continua mais claro, sobretudo em APIs muito genéricas.

Exemplos Comentados (Antes vs Moderno)

Declaração de mapa

Antes

Map<String, List<Integer>> mapa = new HashMap<String, List<Integer>>();

Moderno

Map<String, List<Integer>> mapa = new HashMap<>();

Mesmo tipo, menos redundância.

Coleção com inferência local

Antes

List<String> nomes = new ArrayList<String>();

Moderno

List<String> nomes = new ArrayList<>();

Padrão esperado em código Java moderno.

Erros Comuns e Anti-padr?es

  • Achar que diamond resolve qualquer caso de inferência complexa.
  • Ocultar tipo quando isso dificulta leitura de equipa.
  • Misturar raw types com diamond.

Checkpoint (Mini Quiz)

1. O principal benefício do operador <> é:

  1. Permitir cast implícito inseguro.
  2. Reduzir verbosidade mantendo segurança de tipos. (correta)
  3. Substituir interfaces genéricas.
  4. Eliminar necessidade de compilação.

Explica??o: O ganho é sintático e de manutenção, sem perder tipagem forte.

2. Qual prática deve ser evitada com diamond?

  1. Usar com generics tipados.
  2. Combinar com raw types. (correta)
  3. Aplicar em new HashMap<>().
  4. Usar em listas e mapas comuns.

Explica??o: Raw type elimina garantias de tipo e aumenta risco de runtime errors.

Recap em 5 Pontos

  1. Diamond remove duplicação desnecessária em generics.
  2. Segurança de tipos é mantida.
  3. Nem toda inferência é ideal em contexto complexo.
  4. Legibilidade continua critério principal.
  5. Evita raw types no código moderno.

M?dulo 10 ? 16 min

Java 9 a Java 16: Melhorias Principais

Entre Java 9 e 16 tens melhorias que mudam o dia a dia de equipa. Vais aprender onde está o retorno mais imediato para produção.

Objetivos de Aprendizagem

  • Mapear as melhorias de maior impacto prático entre Java 9 e 16.
  • Decidir quais features adotar primeiro em projetos existentes.
  • Evitar adoções superficiais sem estratégia de compatibilidade.

Explica??o por Blocos

Java 9 e 11: base de plataforma

Java 9: módulos (Project Jigsaw), coleções imutáveis de fábrica e melhorias de API.

Java 11: API HTTP moderna e métodos úteis em String.

Java 10 a 16: ergonomia da linguagem

var para inferência local com cuidado de legibilidade.

Switch expressions e text blocks reduzem ruído em lógica e strings multilinha.

Adoção por valor

Prioriza features com retorno imediato: API HTTP, text blocks, switch expressions.

Mede impacto em clareza de código e manutenção de equipas.

Exemplos Comentados (Antes vs Moderno)

HTTP Client (Java 11)

Antes

URL url = new URL("https://api.exemplo.com");
HttpURLConnection conn = (HttpURLConnection) url.openConnection();
conn.setRequestMethod("GET");

Moderno

HttpClient client = HttpClient.newHttpClient();
HttpRequest req = HttpRequest.newBuilder()
  .uri(URI.create("https://api.exemplo.com"))
  .GET()
  .build();
HttpResponse<String> rsp = client.send(req, BodyHandlers.ofString());

API moderna substitui fluxos antigos verbosos e propensos a erro.

Switch expression

Antes

String nivel;
switch (score) {
  case 1:
  case 2:
    nivel = "Básico";
    break;
  default:
    nivel = "Avançado";
}

Moderno

String nivel = switch (score) {
  case 1, 2 -> "Básico";
  default -> "Avançado";
};

Menos fallthrough acidental e atribuição mais direta.

Erros Comuns e Anti-padr?es

  • Ativar features novas sem rever baseline de versão no build.
  • Usar var em nomes de variável pobres que escondem tipo real.
  • Ignorar impacto de módulos na organização de dependências.

Checkpoint (Mini Quiz)

1. Qual feature pertence ao Java 11 e tem alto impacto prático?

  1. Records finais.
  2. HTTP Client API padrão. (correta)
  3. Virtual threads.
  4. Sealed classes.

Explica??o: A API HTTP tornou o consumo de serviços mais limpo já no Java 11.

2. Qual cuidado é importante ao usar var?

  1. Só funciona com tipos primitivos.
  2. Dispensa compilador com tipos.
  3. Exigir nomes claros para manter legibilidade. (correta)
  4. Substituir todos os tipos explícitos automaticamente.

Explica??o: Inferência não elimina necessidade de código legível.

Recap em 5 Pontos

  1. Java 9–16 trouxe modularidade, ergonomia e APIs modernas.
  2. HTTP Client, switch expressions e text blocks são ganhos concretos.
  3. var deve melhorar leitura, não escondê-la.
  4. Compatibilidade de versão deve ser planeada no build.
  5. Adota por valor real, não por moda.

M?dulo 11 ? 18 min

Java 17 a 21: Features de Alto Impacto

Este bloco é de impacto alto: domínio mais explícito e concorrência mais escalável. É aqui que o Java moderno mostra força total.

Objetivos de Aprendizagem

  • Aplicar records, sealed classes e pattern matching em modelação de domínio.
  • Entender o ganho de virtual threads para workloads concorrentes.
  • Avaliar estratégia de migração para LTS recente.

Explica??o por Blocos

Modelação mais explícita

Records reduzem boilerplate de dados imutáveis.

Sealed classes definem fronteiras de hierarquia de forma controlada.

Pattern matching em switch

switch ganha expressividade para tipos, reduzindo casts manuais e if/else extensos.

Virtual threads

Permitem alto número de tarefas concorrentes com custo mais baixo por thread em cenários I/O-bound.

Não substituem análise de arquitetura, mas simplificam bastante o modelo de concorrência.

Exemplos Comentados (Antes vs Moderno)

Record para DTO imutável

Antes

public final class ClienteDto {
  private final String nome;
  private final String email;

  public ClienteDto(String nome, String email) {
    this.nome = nome;
    this.email = email;
  }

  public String getNome() { return nome; }
  public String getEmail() { return email; }
}

Moderno

public record ClienteDto(String nome, String email) {}

Menos código, contrato de dados explícito.

Pattern matching em switch

Antes

String msg;
if (evento instanceof Erro) {
  msg = ((Erro) evento).mensagem();
} else if (evento instanceof Aviso) {
  msg = ((Aviso) evento).mensagem();
} else {
  msg = "Sem detalhe";
}

Moderno

String msg = switch (evento) {
  case Erro e -> e.mensagem();
  case Aviso a -> a.mensagem();
  default -> "Sem detalhe";
};

Elimina casts repetidos e deixa regras de decisão centralizadas.

Erros Comuns e Anti-padr?es

  • Migrar para virtual threads sem observar limites de I/O, pools e drivers.
  • Usar record em modelos que exigem forte mutabilidade de estado.
  • Ignorar versões de framework que suportam (ou não) features recentes.

Checkpoint (Mini Quiz)

1. Qual feature ajuda a reduzir boilerplate de classes de dados?

  1. Text blocks
  2. Records (correta)
  3. Diamond operator
  4. removeIf

Explica??o: Records encapsulam estrutura de dados imutável com sintaxe compacta.

2. Virtual threads tendem a destacar-se em:

  1. Cargas I/O-bound com muitas operações concorrentes. (correta)
  2. Qualquer cálculo CPU-bound sem limite.
  3. Cenários sem operações bloqueantes.
  4. Substituição direta de todas as bibliotecas de concorrência.

Explica??o: Ganham especialmente quando há muitas tarefas bloqueantes de I/O.

3. Sealed classes servem para:

  1. Aumentar automaticamente throughput de rede.
  2. Controlar quais classes podem estender uma hierarquia. (correta)
  3. Eliminar a necessidade de testes de domínio.
  4. Converter classes em records.

Explica??o: Definem fronteiras de herança, útil para modelação e segurança de domínio.

Recap em 5 Pontos

  1. Java 17–21 melhora modelação de domínio e concorrência.
  2. Records e sealed classes aumentam clareza sem boilerplate.
  3. Pattern matching reduz casting manual.
  4. Virtual threads ampliam concorrência I/O-bound com menor custo.
  5. Migração deve considerar suporte de frameworks e infraestrutura.

M?dulo 12 ? 11 min

Inferência de Tipos com var

Fechamos com maturidade de estilo: var bem aplicado acelera leitura; mal aplicado destrói contexto. Vais sair com critério sólido.

Objetivos de Aprendizagem

  • Aplicar var em contextos onde o tipo é óbvio.
  • Evitar var em locais que exigem explicitação do domínio.
  • Definir guideline de equipa consistente para inferência local.

Explica??o por Blocos

O que var faz de facto

var permite inferência de tipo em variáveis locais com inicialização obrigatória.

Não é tipagem dinâmica: o tipo continua estático em compile-time.

Cenários bons

Inicializações com tipo evidente: new HashMap<String, Integer>(), resposta de método muito claro, loops simples.

Cenários a evitar

Quando a expressão é ambígua ou longa e esconde tipo relevante para entendimento do domínio.

Exemplos Comentados (Antes vs Moderno)

var que ajuda

Antes

Map<String, Integer> stockPorSku = new HashMap<>();

Moderno

var stockPorSku = new HashMap<String, Integer>();

O tipo é claro no lado direito; legibilidade mantém-se.

var que atrapalha

Antes

var resultado = processador.executar(configuracao, contexto, parametros);

Moderno

ResultadoValidacao resultado = processador.executar(configuracao, contexto, parametros);

Quando o retorno não é óbvio, tipo explícito reduz carga cognitiva.

Erros Comuns e Anti-padr?es

  • Substituir tipos explícitos em massa sem revisão de legibilidade.
  • Assumir que var reduz qualidade de tipagem (não reduz).
  • Usar nomes vagos de variável e perder contexto semântico.

Checkpoint (Mini Quiz)

1. Qual afirmação sobre var é correta?

  1. var torna Java dinamicamente tipado.
  2. var só funciona em campos de classe.
  3. var infere tipo local em compile-time. (correta)
  4. var elimina necessidade de inicialização.

Explica??o: A inferência é local e estática, com validação em compilação.

2. Quando preferir tipo explícito em vez de var?

  1. Quando o tipo do retorno não é evidente para quem lê. (correta)
  2. Quando há qualquer uso de collections.
  3. Sempre que estiveres num método curto.
  4. Nunca, porque var é sempre melhor.

Explica??o: O objetivo é legibilidade. Se var esconder informação relevante, usa tipo explícito.

Recap em 5 Pontos

  1. var é inferência local estática, não tipagem dinâmica.
  2. Funciona melhor quando o tipo é evidente no contexto.
  3. Tipo explícito continua essencial em pontos ambíguos.
  4. Guideline de equipa evita inconsistência.
  5. Boa legibilidade supera preferência pessoal de sintaxe.
